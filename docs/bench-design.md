# Bench design — why the harness is shaped this way

The agent benchmark under `benchmarks/timing_diagnosis/` is a TerminalBench-style measurement instrument for chip-design diagnosis. This document explains *why* the design choices are what they are, so future agents adding cases or extending the harness make changes that fit the same shape.

## Two parts, built together

The bench is one half of a two-sided artifact:

- **Side A — the instrument.** `harness.py`, `run.py`, `agent.py`, the task corpus. Frozen, deterministic, measures one specific capability (timing-failure diagnosis at the moment).
- **Side B — the subject.** The agent system (`agent.py` + tool surface + future extensions) doing real EDA work. The product surface.

Each only makes sense in light of the other. A bench without a subject is academic. A subject without a bench is theater.

The harness already enforces this separation in code: `agent.py` and `harness.py` share no state; `agent.py` produces JSON, `harness.py` grades JSON. They communicate through a stable on-disk contract (`hidden_oracle/golden.json`). When extending either side, preserve that boundary.

## Two-tier oracle

Every task's gold answer comes from one of three sources, declared in `task.json` and enforced by the harness:

| `oracle_type` | What it means | When to use |
|---|---|---|
| `physics_first_principles` | Hand-constructed seed fixture with deterministic arithmetic. The agent's diagnosis is verifiable from the report numbers alone. | Harness development. Three seed cases under `physics_*/` exist for this. Don't add more — they're for testing the contract, not measuring agents. |
| `external_tool` | Golden derived from a real EDA tool (OpenSTA, PrimeTime, Genus, Innovus, etc.). | Use when an industry-standard tool can produce the gold answer mechanically. Currently aspirational — needs OpenSTA installed. |
| `external_authority` | Golden sealed by a named human authority (GitHub issue maintainer comment, paper, textbook). | Use when the diagnosis requires expert judgment beyond what a tool can produce. The published `authority_001_ol_1371_hold_repair_setup_conflict` case is the canonical example. |

The two-tier design (tool-grounded + human-grounded) catches different failure modes:

- **Physics / tool oracles catch hallucination on objective facts.** OpenSTA cannot be socially pressured; it disagrees in numbers. An agent that lies about slack arithmetic gets caught immediately.
- **Authority oracles catch "technically right, contextually wrong."** A senior chip designer often says "yes the local cause is X, but the actual fix lives upstream at Y." A physics oracle can't tell the difference between *correct local explanation* and *right architectural answer*.

Both tiers are external to the agent triad. Neither was generated by us reasoning at each other.

## Chamber-check principle

The deepest rule, named explicitly by Mani: when 3+ agents are co-designing a system whose purpose is to *produce truthful output*, agreement among the agents is **not** evidence the output is right. Agreement is just confirmation bias scaled across instances.

This affects every choice in the bench:

- **Authority cases must be sealed by external parties.** No "Claude wrote a plausible diagnosis and Codex agreed it sounds right" — the diagnosis must come from a maintainer comment or a paper, with verbatim quotes preserved in `provenance.md` for audit.
- **Physics cases use deterministic arithmetic.** No "the model probably gets this right" — the answer follows from the report bytes.
- **Tool oracles use industry-standard tools.** OpenSTA's slack number is OpenSTA's slack number, regardless of what any agent thinks it should be.

The full memo is in `AGENTS.md` § *Chamber check*. Future agents writing strategy docs in this repo: re-read it before starting.

## Visible / hidden split

Every task has the same on-disk shape:

```
tasks/<task_id>/
  task.json              # manifest (visible to harness, sets oracle_type)
  prompt.md              # the question (visible to agent)
  input/                 # artifacts the agent inspects (visible)
    ...
  hidden_oracle/         # NEVER visible to the agent
    golden.json          # grader contract (visible to harness, never agent)
    provenance.md        # audit trail (chamber-busting evidence)
```

The agent path sandbox in `agent.py::_safe_input_path` enforces this in code: every file-reading tool resolves through it, and `hidden_oracle/` is unreachable. Path traversal (`../hidden_oracle/...`), absolute paths (`/etc/passwd`), and `input/../oracle/...` all raise `PathSandboxError`.

If you add a tool that reads files (e.g. a future `find` or `cat -n` tool), route it through `_safe_input_path` so the sandbox guarantee survives.

## Schema = the API

The grader walks a fixed JSON contract: `required_exact`, `required_numeric`, `required_evidence`, `required_next_action_terms`. Same shape for physics, tool, and authority oracles. Same agent output shape for every task type.

Why uniform: it lets the same `agent.py` answer every task without per-task plumbing, and the same scorecard report every result. The full schema is documented in `docs/golden-schema.md` — read it before authoring a new `golden.json`.

When research-Claude (or any curation agent) hands over a "rich" answer format (verbatim quotes, fix commits, named authorities) — that goes in `provenance.md`. The `golden.json` stays minimal and grader-shaped. Don't extend the grader to handle richer schemas; the value is in the boundary.

## Why TerminalBench shape

The bench borrows TerminalBench's shape: a fixed corpus of frozen tasks with deterministic grading, scored agent-by-agent for comparison. The reasons:

- **Public reproducibility.** Anyone can clone the repo, run `python benchmarks/timing_diagnosis/run.py grade <task> <answer.json>`, and verify their numbers.
- **Per-model scorecards.** Future cards: `claude-opus-4-7` vs `claude-sonnet-4-6` vs `gpt-5` vs `kimi-k2.5` on the same tasks, same harness, same scoring.
- **Adversarial cases as a category.** Some authority cases (OL-958: misleading source comment; OR-4833: misleading title) test whether agents commit to surface evidence vs. dig past it. Calibration cases (OR-4359: stale-closed feature/bug ambiguity) test whether agents refuse to over-commit when reality is ambiguous. Different agents will handle these differently — that's what the bench reveals.
- **Growth path.** Today: timing diagnosis. Tomorrow: synthesis-failure diagnosis, DRC diagnosis, formal-verification debugging, RTL bug-finding, PPA-tradeoff selection. Each becomes a new directory under `benchmarks/`, same shape.

## What the bench is *not*

- **Not a replacement for ORFS-agent or other PPA-optimization benchmarks.** Kahng's group owns optimization. The bench measures a different capability: *can the agent see, diagnose, and explain*, not *can it search a configuration space*. The two are complementary, not competitive.
- **Not a chat eval.** Open-ended "give me the design intent" prompts aren't here. Every task has a deterministic answer the grader can score.
- **Not a smoke test for `eda-parse`.** That's `benchmarks/workflow_testbench.py`. The agent bench tests *what an agent does with eda-parse*, not whether the parser works.

## Open questions / honest gaps

These are real and not yet resolved. Future agents picking up bench work should know:

- **Real-API smoke test.** Every closed-loop test so far is mock-mode. The first real Anthropic API call against an authority case is the actual moment of truth — does Claude diagnose OL-1371 the way Cherry/Blanchard/Donn diagnosed it? Hasn't fired yet. See `docs/PLAN.md` § *Open work* for the gating decision on when to spend the API budget.
- **OpenSTA tier.** The `external_tool` oracle type is declared in the schema but no case yet uses it. Wiring requires installing OpenSTA on the box and writing a converter from OpenSTA reports to grader-shaped goldens. Real but not done.
- **Beyond timing diagnosis.** All current cases are timing failures. The bench should grow to UVM testbench debugging, DRC violations, synthesis failures, and PPA tradeoffs. Same shape, new directories.
- **Adversarial scoring rubric.** Calibration cases like OR-4359 require the agent to *refuse to commit*. The current grader rewards exact-match answers — it doesn't yet have a "this answer is appropriately hedged" rubric. Open design question.
- **Cost budget per task.** Real API runs spend tokens. No infrastructure yet for capping per-case spend or for batching runs efficiently across multiple models. `agent.py`'s `--max-iters` is the only knob today.

If you're picking one of these up, document your approach in `docs/dev-log.md` as you go.
